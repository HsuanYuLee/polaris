# Epic Description 結構化模板

供 `refinement`（Phase 1 整合）和 `epic-breakdown`（Step 13 整合）使用。
也可分享給 PM 作為開單參考。

## 模板結構

```markdown
## 背景與目標

<!-- 為什麼做這個？解決什麼問題？商業目標或使用者痛點 -->

## Scope

### 做什麼
- ...

### 不做什麼（Out of scope）
- ...

## Baseline（若有量化數據）

<!-- 效能、錯誤率、轉換率等現況數據，作為改善前的基準 -->

| 指標 | 現況 | 目標 |
|------|------|------|
| ... | ... | ... |

## 瓶頸分析（若為效能/優化類）

<!-- 根據數據分析的主要瓶頸，附來源（Lighthouse、DevTools 等） -->

## AC（Acceptance Criteria）

1. ...
2. ...
3. ...

## 開發驗收（RD 自驗）

<!-- Lighthouse 跑分、DevTools Coverage、before/after 比較等 RD 層級的驗收方式 -->

## 可行做法

<!-- 技術方案概述，不需詳細到 code level，讓 RD 拆單時有方向 -->

### 方案 A: ...
- 做法：...
- 預估影響：...

### 方案 B: ...（若有替代方案）

## 拆單總覽

<!-- 拆單完成後由 epic-breakdown 自動填入 -->

| # | Key | Summary | Points |
|---|-----|---------|--------|
| 1 | ... | ... | ... |
| **Total** | | | **N 點** |

## 依賴

- [ ] ...（依賴的其他 ticket、API、DS 改動等）

## 待確認

- [ ] ...（未定事項、需 PM 回覆的問題）

## 參考資料

- Figma: ...
- API doc: ...
- PRD: ...
```

## Readiness Checklist

Refinement 開始前或結束時，用此 checklist 評估 Epic 完整度：

| # | 項目 | 說明 | 必要性 |
|---|------|------|--------|
| 1 | **背景與目標** | 為什麼做？商業價值或使用者痛點 | 必要 |
| 2 | **AC** | 可驗收的條件，至少 3 條 | 必要 |
| 3 | **Scope** | 涉及哪些頁面/元件/API，含 out of scope | 必要 |
| 4 | **Figma / 設計稿** | UI 相關才需要 | 視情況 |
| 5 | **API 文件** | 有串接才需要 | 視情況 |
| 6 | **Edge cases** | 空狀態、錯誤處理、loading、多語系 | 建議有 |
| 7 | **依賴** | 跨團隊、跨專案的前置條件 | 建議有 |
| 8 | **Baseline 數據** | 效能/優化類必要，功能類可選 | 視情況 |

**判斷標準：**
- 必要項全部 ✅ → 可進入估點/拆單
- 有 ⚠️ 或 ❌ → 列出缺項，tag PM 補充
- 效能/優化類額外需要 Baseline + 瓶頸分析

## 給 PM 的開單建議

> 這段可直接分享給 PM，幫助他們開出更完整的 Epic。

### 好的 Epic 長什麼樣？

1. **背景不只是「做 X 功能」**：說明為什麼要做、解決什麼問題、影響多少使用者
2. **AC 是可驗收的**：「做好商品列表」❌ →「使用者能依價格排序，結果在 500ms 內顯示」✅
3. **Scope 有邊界**：明確列出「不做什麼」，避免 scope creep
4. **附上參考資料**：Figma、API doc、競品截圖等，減少 RD 猜測時間
5. **量化指標**（若適用）：效能提升多少、錯誤率降到多少

### 常見問題

| 問題 | 影響 | 建議 |
|------|------|------|
| 只有標題沒有 description | RD 無法估點，refinement 耗時 | 至少填背景 + AC |
| AC 太模糊 | 開發方向可能偏離 PM 期望 | 用「使用者做 X → 看到 Y」格式 |
| 缺 Figma 連結 | RD 無法判斷 UI 複雜度 | 即使是草圖也好過沒有 |
| 沒有 out of scope | 開發到一半發現範圍比預期大 | 列出「本次不做」的項目 |
