## Refinement Input Requirements

Epic 進入 breakdown 前，refinement 階段必須確認：

### 目標驗證頁（mandatory）

Epic 必須附上**目標驗證頁清單** — 至少每個影響的頁面類型各一頁，且必須在 SIT 上可達（HTTP 200 + 正常渲染）。

| 欄位 | 說明 | 範例 |
|------|------|------|
| `page_type` | 頁面類型 | homepage, product, destination, category |
| `url` | SIT 上的完整 URL | `https://sit.example.com/zh-tw/product/133300` |
| `status` | 確認可達 | 200 OK, renders correctly |

**範例（PROJ-483）：**

```
目標驗證頁：
- homepage: /zh-tw/ (200 ✓)
- destination: /zh-tw/destination/jp-japan (200 ✓)
- category: /zh-tw/category/global/sightseeing-tours/list (200 ✓)
- product: /zh-tw/product/133300 (200 ✓)
```

**Why:** 沒有目標頁就無法錄 fixture、無法跑 VR、無法寫驗收單。目標頁是整個驗證流程的輸入。

**Who provides:** PM 在 Epic description 中列出，RD 在 refinement 確認可達性。如果 PM 沒列 → refinement 階段補問。

## Fixture Lifecycle (per-Epic)

Fixtures 跟著 Epic 生命週期：建立 → 開發期間使用 → Epic 完成後刪除。

### 穩定測資單（Fixture Recording Ticket）

Epic breakdown 時，在 **API 調整單（如果有）之後、實作單之前**，建立一張「穩定測資」ticket：

```
breakdown 順序：
  1. API 調整單（如果有）
  2. 📦 穩定測資單          ← 這張
  3. 實作單 × N
  4. 驗收單（per-AC）
```

**任務內容：**

#### Step 1: AI Discovery（API Endpoint 探索）

AI 探索所有相關 repo，找出 API endpoint 的定義方式和 routing 設定。這一步只做一次，產出 `proxy-config.yaml`。

**為什麼用 AI 而不是腳本：** API endpoint 的定義方式是 project-specific 的（有的用 .env vars、有的藏在 webapi code 裡、有的在 config file）。確定性腳本無法覆蓋所有格式，AI 讀 code 理解架構更可靠。

**Discovery 流程：**
1. 掃描所有相關 repo 的 API endpoint 定義：
   - .env / .env.template → 找 `*_API_BASE_URL`, `*_API_URL` 等 pattern
   - nuxt.config / app config → 找 runtime config 中的 API URL
   - 原始碼（webapi routes, HTTP client config）→ 找 hardcoded service URLs
2. 比對 Mockoon env 的 `proxyHost` → 建立 routing mapping
3. 產出 `proxy-config.yaml` → 存到 `{EPIC_KEY}/`

**產出格式：**

```yaml
# proxy-config.yaml — Generated by Polaris during fixture recording
# Epic: PROJ-483
# Generated: 2026-04-05
# Source repos: acme-web-app (master), acme-member-api (master)

routes:
  - target_host: "dev.example.com"
    mockoon_port: 4001
    env_override: "PHP_API_BASE_URL=http://localhost:4001"
    discovery: "env_var"            # AI 怎麼找到的
    source_file: "apps/main/.env"   # 來源檔案

  - target_host: "recommend.sit.example.com"
    mockoon_port: 4004
    env_override: "RECOMMEND_API_BASE_URL=http://localhost:4004"
    discovery: "env_var"
    source_file: "apps/main/.env"

  - target_host: "api-product.sit.example.com"
    mockoon_port: 9999
    env_override: "PRODUCT_API_BASE_URL=http://localhost:9999"
    discovery: "code_scan"
    source_file: "src/webapi/product/config.ts"
```

`discovery` 和 `source_file` 欄位記錄推理路徑 — 下一個 Epic 開工時 AI 可以先參考，加速 discovery。

**跨 Epic 知識累積：** 第一次 discovery 最慢（AI 讀 code 理解架構）。結果寫在 proxy-config.yaml 和 memory 中，同 repo 的下一個 Epic 可以重用推理路徑。

#### Step 2: 啟動環境

1. `polaris-env.sh start {company} --vr --epic {EPIC_KEY}`
2. Script 讀 `{EPIC_KEY}/proxy-config.yaml` → 提取 env overrides
3. 啟動 Mockoon from `{EPIC_KEY}/` (proxy mode)
4. 啟動 dev server（帶上 proxy env vars，process-scoped override）

**設計原則：** AI 負責生成 proxy-config.yaml（discovery），script 負責讀取和執行（deterministic）。符合 Deterministic Enforcement Principle — AI 做判斷，script 做強制。

#### Step 3: 錄製

1. Purge Mockoon transaction logs
2. 用 Playwright 掃過所有目標驗證頁（來自 Refinement Input）
3. Harvest Mockoon transaction logs（proxied requests）
4. 轉換為 static Mockoon routes → 寫入 `{EPIC_KEY}/*.json`

#### Step 4: 驗證

1. 切 Mockoon 到 mock mode（非 proxy）— 重啟 without --proxy
2. 重新掃目標頁
3. 確認全部正常渲染（HTTP 200 + 關鍵元素存在）

如果驗證失敗 → 回到 Step 3 補錄遺漏的 routes。

**Folder 結構：**

```
specs/{EPIC_KEY}/tests/mockoon/    # Per-epic fixtures (see references/epic-folder-structure.md)
├── dev.example.com.json           # 主要 fixture（錄製的 API responses）
├── api-lang.sit.example.com.json  # 其他 service fixtures
└── ...

{company_base_dir}/mockoon-config/ # 跨 Epic 共用 config
├── proxy-config.yaml
└── demo.json (optional)
```

**前置條件：** API 調整單已完成（如果有）。這確保錄到的 fixture 包含了新 API 的 response。

**沒有 API 調整的 Epic：** 直接錄 SIT 現有的 response。流程不變，只是少一張前置單。

**全新 repo clone：** 不依賴 pre-existing .env.local。AI Discovery 從零探索，proxy-config.yaml 是 Epic 產物，不是 repo 常駐設定。

### 開發期間

- 所有實作單基於 Epic fixture 開發
- 每張 task PR（→ feat branch）必須通過 VR strict mode
- `mockoon-runner.sh` 指向 `{EPIC_KEY}/` folder

### 整合驗證

每次 task merge 回 feat branch 後，跑完整驗證：
- 所有 unit tests
- VR（全目標頁）
- 所有已整合子單的驗證項（Layer 3，見下方）

### Epic 完成 — Fixture Cleanup

Feature PR merge 到 develop 後：
1. 刪除 `specs/{EPIC_KEY}/tests/mockoon/` folder（或保留作為 archive — 見 `references/epic-folder-structure.md`）
2. 更新 workspace-config 移除 active fixture set 指向

**象徵意義：** fixture folder 的刪除代表這個功能或 bug 完整做完了。

### Stale Fixture Hygiene

Fixture folder 超過 30 天未更新 + Epic status 不是 In Development → 標記為 stale，提醒刪除。可在 `converge` 或 `epic-status` 中 check。

### VR 判定模式：Figma Match vs Strict

改變外觀的 task（有 Figma 設計稿）和不改外觀的 task，使用不同的 VR 判定邏輯：

| Task 有 Figma link？ | VR 模式 | 判定標準 |
|----------------------|---------|---------|
| 有 | **Figma Match Mode** | 渲染結果 vs Figma 設計稿結構吻合 → PASS（忽略文案差異） |
| 沒有 | **Strict Mode** | vs Epic fixture baseline zero-diff → PASS |

**Figma Match Mode 流程：**

1. `get_metadata(figma_node)` → 讀 frame width（如 1440px）
2. `get_screenshot(figma_node)` → Figma 設計截圖
3. `Playwright(viewport={ width: figma_width })` → 渲染截圖（viewport 自動匹配 Figma frame 寬度，避免 RWD 干擾）
4. AI Vision checklist 比對 → structured JSON output：
   - 主要區塊數量一致？
   - 區塊上下排列順序一致？
   - 每個區塊內元件類型一致？（圖片、按鈕、卡片列表）
   - 有沒有 Figma 有但頁面缺少的區塊？
   - 有沒有頁面有但 Figma 沒有的區塊？
5. `blocks_match: true` → PASS；`blocks_match: false` → FAIL（列出 missing/extra）

**Refinement 品質規則：** 改變畫面的 task 必須附 full-page Figma link（含版位上下文）。純 component Figma 僅限 design system 類 ticket。

**Status: 待 PoC 驗證。** 需要一張有 Figma 設計稿的 task 來試跑 AI Vision 比對品質。PoC 通過 → 寫進 VR skill 作為 `figma_match` mode。不通過 → 退回 per-task 影響頁面標記（方案 A）。

### Feature PR 前真實 API 驗證

開發期間 fixture 完全隔離真實 API，不需要定期 re-record。

**風險只在 merge 時實現：** Feature PR merge 到 develop 後，fixture 刪除，真實 API 接手。如果其他團隊在 Epic 期間改了 API response 結構，頁面可能壞。

**解法：Feature PR 前用真實 API 跑一次 VR。**

```
Feature PR 前：
  1. rebase develop
  2. 關閉 Mockoon（不用 fixture）
  3. 用真實 SIT API 跑 VR
  4. 比對結果：
     - 跟 fixture VR 一致 → 安全，無 API 變更
     - 有差異 → 標記「API 已變更」，回報 PM 評估
```

**責任邊界：** 嚴守開工規範 + merge 前 catch 差異 = 仁至義盡。其他團隊的 breaking change 不是我們的 regression，發現後回報 PM 決定改、適配、或重做。不需要 re-record 機制。

## Three-Layer Verification

| Layer | Purpose | Timing | Form |
|-------|---------|--------|------|
| **Task test plan** | PR 品質紀錄，留存讓人 review | task branch → feature PR | Sub-task（紀錄用，在 PR gate 內，非獨立驗收） |
| **Epic per-AC verification** | 業務目標達標證明 | 所有 task merge 回 feature 後 | TEAM Task × N，Playwright E2E 逐條跑 |
| **Feature integration test** | task 之間互動沒問題 | 同上 | 驗收單之一或獨立一張 |

## Layer 1: Task Test Plan (sub-task)

**What:** 每張實作 task 開一張 sub-task 記錄測試計劃。

**Purpose:** 不是驗收 gate，而是品質紀錄。讓其他人 review 這個 Epic 時，能確認每張 task 是如何通過框架定下的測試計劃的。未來可以與時俱進增強或調整。

**Content:**
```
測試計劃 — {TICKET_KEY}

1. 品質檢查：lint ✓ / test ✓ / coverage ✓
2. VR：{triggered / not applicable}（理由）
3. 驗證：{engineer-delivery-flow Step 3 行為驗證結果 or 手動驗證步驟}
4. PR review：{approved by X}
```

**Trigger:** `engineering` 或 `breakdown` 建立 task 時自動建 sub-task。engineer-delivery-flow Step 2（Local CI Mirror）和 Step 3（Behavioral Verify）跑完後更新內容。

**Not a blocker:** task 的 PR merge 不依賴 sub-task 狀態。sub-task 是事後記錄。

## Layer 2: Epic per-AC Verification (TEAM Task)

**What:** Epic 的每個 AC 開一張驗收 ticket。

**Source of AC items:**
- QA/PM 提供的 happy flow 和驗收標準
- RD 補充 Polaris 推斷的重要檢查（VR、效能指標、邊界條件）

**Size threshold:**
- Epic > 8 pts 或 > 2 tasks → per-AC 拆驗收單
- Epic ≤ 8 pts 或 ≤ 2 tasks → 合併成一張「Epic 驗收」

**When to create:**
- `breakdown` 拆單時自動在最後產生
- Status 設為 Open / Waiting for Development
- 標記依賴：depends on 所有實作單

**When to run:**
- `epic-status` 偵測所有實作單 PR merged 回 feature branch 後提醒
- 驗收前先 rebase develop（確保 feature branch 是最新的）

**Environment tagging:**
每張驗收單標記可在哪個環境驗：

| Tag | Meaning | Example |
|-----|---------|---------|
| `env:feature` | 可在 feature branch 驗 | VR zero-diff, code structure check |
| `env:stage` | 需要 stage 環境 | 真實 TTFB 數字, 真實 API response |
| `env:both` | feature 初步驗 + stage 最終驗 | 效能指標（local 初步, stage 最終） |

同一張驗收單可跑兩次：feature branch 初步通過 → stage 最終通過。更新結果在 ticket comment。

**Execution:**
- Playwright E2E 逐條跑，結果更新在各驗收 task 上
- 結果作為 evidence 給 QA/PM 確認標準達到

## Layer 3: Feature Integration Test

**What:** feature branch 每次整合新的 sub-task code 後，重跑**所有已整合子單**的驗證。不是只在「全部 merge 完」才跑一次。

**Why:** PROJ-483 試跑實證 — TEAM-3461 和 TEAM-3556 各自驗過，但 rebase 整合時 conflict resolution 把正確的 i18n endpoint 蓋掉，個別驗證都沒重跑，直到上線前才發現翻譯全壞。

**Core rule: 有整合就要驗，不管整合兩個或三個 task。**

### Fixture Lifecycle (per-Epic)

Layer 3 的 determinism 依賴 Mockoon fixtures。Fixtures 跟著 Epic 生命週期：

1. **Epic 開始** — 在 develop branch 上錄 fixtures（Playwright 開所有頁面，Mockoon proxy mode 捕捉 API response）
2. **API task 完成後** — 若 Epic 有 cross-repo API 變更，re-record fixtures（breakdown 的 API-first 排序確保這發生在前端 task 之前）
3. **前端開發期間** — 所有整合測試基於穩定 fixtures，VR strict mode（zero-diff）
4. **Epic release** — fixtures 消滅，下個 Epic 重新錄

沒有 fixtures 時（Epic 間空窗期、fixtures 錄製前），Layer 3 仍可執行但切換為 **non-deterministic mode** — 結果可能包含假陽性。

> 公司層具體操作見 `playwright-testing.md` § Mockoon Fixture Lifecycle

### Trigger Points

整合驗證不是單一觸發點，而是多層觸發同一個動作：

| 觸發場景 | 時機 | 信心度 |
|----------|------|--------|
| Polaris 幫 merge task → feat | 整合完立刻跑 | 最高（確定性） |
| 手動 merge 後使用者說「繼續」/「next」 | my-triage Step 0 Resume scan 偵測 feat branch 有新 commit | 高 |
| 跑 epic-status / converge | 推進前兜底 | 中（可能距離整合已有時間差） |
| 開 feature PR 到 develop | git-pr-workflow quality gate | 兜底（最後防線） |

四層觸發同一個動作：收集 feat branch 上所有已整合子單的驗證項，全部跑一輪。

### Executable Verification Format

每張子單的驗證項必須是**可執行的**，不能只是 JIRA 文字描述。建立子單時，驗證項需包含至少一個 executable check：

```yaml
# 驗證項格式（寫在 JIRA sub-task description 或 comment）
verification:
  # 首選：Playwright 瀏覽器驗證（有 session、有渲染、有 JS 執行）
  # {BASE_URL} 由公司層 playwright-testing reference 定義（如 dev.example.com）
  - name: "商品頁 SSR 渲染正常"
    type: browser
    command: |
      npx playwright test --grep "product-page-render"
      # 或 inline script:
      # page.goto('{BASE_URL}/zh-tw/product/12345')
      # expect(page.locator('[data-testid="product-title"]')).toBeVisible()
    assert: "page renders without error, key elements visible"

  - name: "i18n 翻譯正常（瀏覽器）"
    type: browser
    command: |
      # page.goto('{BASE_URL}/zh-tw/product/12345')
      # expect(page.locator('text=加入購物車')).toBeVisible()
    assert: "translated text renders correctly in browser context"

  # API-only 驗證（無渲染需求時才用 curl）
  - name: "footer cache header"
    type: endpoint_check
    command: "curl -sI {BASE_URL}/api/_nuxt/footer"
    assert: "headers contains 'x-nitro-cache'"

  - name: "TTFB < 1000ms"
    type: performance
    command: "curl -o /dev/null -s -w '%{time_starttransfer}' {BASE_URL}/zh-tw/"
    assert: "time_starttransfer < 1.0"

  - name: "SSR 無 waterfall"
    type: log_check
    description: "dev server log 不應出現 sequential fetch warning"
```

| Type | 自動化程度 | 說明 |
|------|-----------|------|
| `browser` | 全自動 | **首選** — Playwright 瀏覽器驗證（有 session、有渲染） |
| `endpoint_check` | 全自動 | curl + response assertion（僅限無渲染需求的 API） |
| `performance` | 全自動 | timing assertion |
| `log_check` | 半自動 | 需人工判讀 log |
| `visual` | 半自動 | VR 截圖比對 |
| `manual` | 手動 | 標記步驟，人工執行 |

### Execution Flow

```
feat branch 整合了新的 sub-task code
  ↓
1. 列出所有已整合的 sub-task（git log 比對）
2. 從 JIRA 收集每張的 verification items
3. 起 dev server on feat branch
4. 逐項執行驗證
5. 輸出報告：
   TEAM-3461 i18n 翻譯    ✅ PASS (data keys: 1847)
   TEAM-3461 SSR parallel  ✅ PASS (no waterfall)
   TEAM-3462 footer cache  ✅ PASS (x-nitro-cache: HIT)
   TEAM-3556 prefetch      ✅ PASS (prefetch triggered)
   ─────────────────────────
   Integration: 4/4 PASS
6. 任一 FAIL → 標記「整合回歸」，阻擋推進
```

### Content (unchanged)
- VR 截圖比對（feature branch vs main/develop baseline）
- E2E smoke test（key user flows）
- 如果 Epic 涉及效能 → 效能基準對比

**Form:** 可以是驗收單之一（如「Feature 整合 VR」），或獨立一張。

## PROJ-483 Lessons Learned (Epic #1 Trial)

PROJ-483 整合測試試跑產出的具體教訓，已驗證並固化在本文件中。

### 1. 瀏覽器優先，curl 退居 API-only

**問題：** curl 打 endpoint 沒有 session、沒有 cookie、沒有 JS 執行。頁面是否真的渲染正常、翻譯是否正確顯示、CSR hydration 是否成功 — curl 一概看不到。

**規則：** 整合測試的 `type` 首選 `browser`（Playwright）。只有純 API header 檢查（如 cache header）或 timing 測試才用 `endpoint_check`。

### 2. URL 格式規範

your-app 的 URL 有嚴格的格式要求，寫錯會 404：

| 欄位 | ✅ 正確 | ❌ 錯誤 |
|------|--------|--------|
| locale | `zh-tw`（小寫） | `zh-TW` |
| location | `tw-taiwan`（urlName） | `A01-001`（area code） |
| 完整範例 | `/zh-tw/product/12345` | `/zh-TW/product/12345` |

**來源：** SIT 站的實際 URL 是 source of truth，直接從 SIT 複製再換 host。

### 3. 測資來源：SIT → localhost

**SIT DB 和 dev DB 是同一個。** SIT 有的商品、locale、location，dev 環境就有。

**正確做法：**
1. 到 SIT 站 (`https://sit.example.com`) 取目標頁面 URL
2. 記下 URL pattern（locale、location、商品 ID 等）
3. 替換 host 為 `localhost:3001`
4. 在 Playwright 裡直接 `page.goto(localUrl)`

**不要：** 自己猜商品 ID、猜 URL path、或從 DB 撈 ID。SIT 站上看得到的就是可用的測資。

### 4. 驗證項選擇

| 好的驗證項 | 不好的驗證項 |
|-----------|------------|
| 頁面是否渲染（Playwright 截圖 + 元素斷言） | curl 打 API 看 JSON |
| 翻譯文字是否出現在畫面上 | i18n API response 有幾個 key |
| 關鍵互動是否可用（按鈕、連結） | HTTP status code 200 |
| SSR hydration 完成（無 console error） | 無渲染的 API timing |

## Feature Branch Flow

```
feature/{EPIC_KEY}-{description}
  │
  │  ── 前置階段 ──
  ├── API 調整單（如果有）→ PR to feature branch
  ├── 📦 穩定測資單 → 錄製 specs/{EPIC_KEY}/tests/mockoon/
  │
  │  ── 實作階段（基於固定測資）──
  ├── task/{TICKET_KEY}-{desc} → PR to feature branch（必須通過 VR）
  │     └── sub-task: 測試計劃（Layer 1, 品質紀錄）
  ├── task/{TICKET_KEY}-{desc} → PR to feature branch（必須通過 VR）
  │     └── sub-task: 測試計劃
  ├── ...（每次整合回 feat → 跑全部測試 + 已整合子單驗證）
  │
  │  ── 驗收階段（所有 task merge 回 feature 後）──
  │  ── Step 0: rebase develop ──
  ├── {TICKET_KEY} — 驗收 AC1 (Layer 2)
  ├── {TICKET_KEY} — 驗收 AC2 (Layer 2)
  ├── {TICKET_KEY} — Feature 整合測試 (Layer 3)
  │
  │  ── PR 階段 ──
  ├── feature → develop PR（VR Gate 必須 PASS）
  │

  └── PR merged → cleanup specs/{EPIC_KEY}/tests/mockoon/ (optional archive)
```

## VR Gate (git-pr-workflow integration)

Feature PR 到 develop 前，git-pr-workflow 新增 **Step 3b: VR Gate**。

### 觸發條件

兩項同時滿足：
1. workspace-config 有 `visual_regression.domains[]`
2. changed files 包含前端可見代碼（`pages/`, `components/`, `layouts/`, `composables/`, `*.vue`, `*.css`, `*.scss`, `server/`, `plugins/`）

兩者都滿足 → VR gate 啟動。否則 skip。

### Gate 行為

| VR 結果 | 動作 |
|---------|------|
| `PASS` | 繼續開 PR |
| `PASS_WITH_DIFFS` | 顯示 diff，問使用者是否接受（可能是預期改動） |
| `BLOCK` | 停止，必須修復後重跑 |

### 在 pipeline 中的位置

```
git-pr-workflow / engineering:
  Step 1-2: branch, simplify, Local CI Mirror (engineer-delivery-flow § Step 2)
  Step 3.5: VR Gate（條件觸發，呼叫 visual-regression skill）
  Step 4-8: review, rebase, commit, PR, JIRA...
```

VR Gate 在 Local CI Mirror 之後、commit 之前。這樣 VR fail 時還沒 commit，可以直接修。

## Edge Cases

| Situation | Handling |
|-----------|---------|
| Epic 只有 1 張 task | 合併所有驗證為一張，不拆 per-AC |
| AC 只能在 stage 驗 | 標記 `env:stage`，feature branch 階段跳過，stage deploy 後跑 |
| 驗收跑出非預期差異 | 評估是否開 fix ticket 或接受差異，記錄在驗收 ticket comment |
| Feature branch 跟 develop 衝突 | rebase + 解衝突後重跑整合測試 |
| 某張 task PR 一直沒 merge | `epic-status` Phase 2 路由 → engineering revision mode 或催 review |
| QA/PM 沒提供 happy flow | RD 根據 Epic AC 自擬，標記「RD 推斷，待 PM 確認」 |
